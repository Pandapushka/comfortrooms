# Отчет фронтового тестирования Comfort Rooms

Дата начала проверки: 2026-05-31.

Цель: проверить сайт как пользователь и администратор через HTTP/front-сценарии, не исправляя найденные ошибки до составления списка.

## Правила этого раунда

- Сначала проверка, потом список ошибок.
- Исправления в код не вносить в рамках тестового прохода.
- Для админских сценариев использовать отдельную тестовую SQLite-базу, чтобы не менять текущую рабочую учетную запись администратора.

## Что будет проверено

- Публичные страницы.
- Форма заявки на странице `Изготовление под заказ`.
- Форма заявки на странице `Контакты`.
- Вход администратора.
- Доступ к админке.
- Редактирование текстовых блоков через админку.
- Загрузка изображения через админку.
- Отображение загруженного изображения в публичной карусели страницы `Изготовление под заказ`.
- Замена/редактирование/удаление изображения, если базовый сценарий загрузки пройдет.

## Результаты

Проверка в процессе. По изображениям уже зафиксирован важный вывод: дефолтные картинки должны оставаться только fallback-вариантом, а загруженные через админку изображения должны иметь приоритет на публичных страницах.

## Исправлено после проверки

- Добавлена общая логика получения `PageImages` по slug страницы. Публичные страницы используют первое админское изображение как приоритетное, а дефолтное локальное изображение — только как fallback.
- Для страницы `/sotrudnichestvo/na-zakaz` дефолтная галерея теперь задается в коде и показывается только если в админке нет изображений.
- Старые временные URL `image.qwenlm.ai`, которые могли остаться в dev-БД из seed-данных, нормализуются в локальный fallback `/images/fallbacks/luxury-chandelier-interior.png`.
- Страница `/sotrudnichestvo/na-zakaz` переписана без внешних CDN `tailwindcss`, `fontsource` и `lucide`.
- В админке текстов значения в `<textarea>` выводятся без HTML-сущностей.
- Добавлен публичный маршрут `/sotrudnichestvo` как обзорная страница направлений сотрудничества.
- Главная, контакты, страница `На заказ` и страницы сотрудничества получили первые недостающие блоки из анализа первоисточника.

## Наблюдение пользователя

- На другом компьютере после клонирования проекта из Git изображение успешно добавляется через админку и видно в админке, но не отображается на публичной странице, где должно появиться. Это нужно проверить отдельно по цепочке:
  - файл физически сохраняется в `wwwroot/uploads`;
  - в `PageImages.ImageUrl` записывается корректный публичный путь;
  - публичная страница читает изображения именно для нужного slug;
  - Razor-разметка карусели рендерит новые изображения;
  - статические файлы доступны по URL из `ImageUrl`.
- Страница `/sotrudnichestvo/na-zakaz` долго загружается в браузере. Нужно отдельно проверить, связано ли это с внешними CDN (`tailwindcss.com`, `fontsource`, `unpkg/lucide`) или с тяжелыми внешними изображениями.

## Предварительно найденные проблемы

1. Страница `/sotrudnichestvo/na-zakaz` в браузере может грузиться долго из-за внешних ресурсов. Серверный HTML отдается быстро (`~0.003s`), локальный uploaded-файл тоже отдается быстро. В HTML страницы остаются внешние зависимости:
   - `https://cdn.tailwindcss.com`
   - `https://api.fontsource.org`
   - `https://unpkg.com/lucide@latest`
   - несколько изображений `https://image.qwenlm.ai/...`
2. В админке текстов значения в `<textarea>` отображаются HTML-сущностями вида `&#x421;...`, а не нормальным русским текстом. Это ухудшает редактирование и может привести к сохранению сущностей вместо текста.
3. Страница `Сотрудничество` как отдельный маршрут `/sotrudnichestvo` в текущем проекте не проверялась как публичная полноценная страница. В анализе первоисточника она есть как обзор/список дочерних разделов, а в нашем проекте основной фокус пока на дочерних страницах.
4. Загруженные локально изображения лежат в `wwwroot/uploads`, а эта папка исключена из Git. Поэтому изображения, загруженные на одном компьютере, не появятся на другом компьютере после `git pull`. На другом компьютере нужно либо загрузить изображения заново, либо переносить `wwwroot/uploads` отдельно, либо в будущем подключить общее хранилище файлов.
5. На части публичных страниц изображения, вероятно, остаются вшитыми прямо в Razor/HTML и не проверяют наличие `PageImages` из админки. Из-за этого админское изображение может сохраняться и быть видимым в админке, но не заменять картинку на публичной странице.

## Проверка наблюдения пользователя по картинкам

В тестовой среде загрузка через админку на странице `sotrudnichestvo-na-zakaz` сработала полностью:

- `POST /admin/pages/sotrudnichestvo-na-zakaz/images` вернул `302`.
- В БД появилась запись `PageImages` с `ImageUrl=/uploads/sotrudnichestvo-na-zakaz/...png`.
- Файл физически появился в `wwwroot/uploads/sotrudnichestvo-na-zakaz`.
- URL файла вернул `200`.
- Публичная страница `/sotrudnichestvo/na-zakaz` содержала новый `<img src="/uploads/...">` и title изображения в карусели.

Вывод: базовая цепочка загрузки и отображения работает. Если на другом компьютере загруженное изображение видно в админке, но не видно на публичной странице, наиболее вероятные причины:

- страница открывается из другой БД/другого запуска, чем админка;
- браузер показывает кешированную публичную страницу;
- приложение запущено не из корня проекта и `wwwroot/uploads` пишется/читается из другого `ContentRootPath`;
- картинка добавлена не к slug `sotrudnichestvo-na-zakaz`, а к другой странице;
- ожидается перенос изображений через Git, но `wwwroot/uploads` не хранится в Git.

Дополнительный вывод после проверки: часть публичных изображений сейчас захардкожена прямо в Razor/HTML-коде страниц. Поэтому изображение, добавленное через админку, может сохраниться в БД и отображаться в админке, но не появиться на публичной странице, если конкретное место фронта еще не подключено к `PageImages`.

## Требование к исправлению изображений

Нужно изменить логику публичных страниц так, чтобы изображения из админки имели приоритет над дефолтными изображениями, вшитыми в код.

Правило:

- Если для страницы в админке нет ни одного изображения, публичная страница показывает дефолтные изображения из кода.
- Если для страницы в админке есть одно или несколько изображений, публичная страница показывает изображения из админки.
- Если администратор удаляет все изображения страницы, публичная страница снова возвращается к дефолтным изображениям.

Это правило нужно применить не только к карусели страницы `/sotrudnichestvo/na-zakaz`, где админские изображения уже работают, но и к другим публичным зонам с изображениями:

- hero-изображение главной страницы;
- изображения страницы `О компании`;
- изображения страницы `Контакты`;
- изображения страниц сотрудничества;
- фоновые/иллюстративные изображения, если они должны управляться из админки.

Рекомендуемая техническая модель:

- Для каждой публичной страницы получать `PageImages` по slug.
- В Razor/viewmodel иметь список дефолтных изображений.
- При рендеринге использовать:
  - `adminImages`, если список не пустой;
  - иначе `defaultImages`.
- Для одиночных изображений использовать первое админское изображение как замену дефолтному.
- Для галерей использовать весь список админских изображений, если он не пустой.
